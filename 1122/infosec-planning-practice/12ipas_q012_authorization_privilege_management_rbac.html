iPAS 資訊安全規劃/防護 - 考題互動式教學

題號：12
題型：複選題

題目：「關於權限管理的說明，下列哪些正確？（複選）」

(A) 一旦使用者通過身份驗證後，系統會根據其授權層級與角色來分配相應的權限
(B) 可以確保每個使用者僅能存取其需要的資源或功能
(C) 能夠防止越權存取
(D) 可以透過以角色為基礎的存取控制（Role-based access control，RBAC）實作
【第一步：找關鍵字】

請注意題目中的以下字詞，它們是解題的核心線索：

  • 權限管理（Privilege Management）：本題的主題，即決定誰能做什麼的機制。
  • 身份驗證後：這是時間點的關鍵，先確認身份，再給予權限。
  • 授權層級/角色：這是分配權限的依據。
  • RBAC（Role-Based Access Control）：這是目前業界最主流的權限管理方法。
  • 越權（Unauthorized Access）：權限管理主要想防範的風險。
【第二步：白話翻譯】

題目其實在問：

「當一個人登入電腦系統後，系統是怎麼決定他可以開哪些檔案、不能開哪些檔案的？這個機制有什麼好處？常見的做法是什麼？」

簡單來說，就是在考你懂不懂「系統怎麼管鑰匙」。

【第三步：解題思路】

這題在考什麼觀念？

主要考核：身份驗證 (Authentication) 與 授權 (Authorization) 的區別，以及 最小權限原則 和 RBAC 機制 的應用。

原理與規則（生活比喻）

想像你要進入一間有很多管制區域的大公司：

  1. 身份驗證 (Authentication)：你在大門口刷門禁卡，警衛確認「你是誰」（例如：你是員工王小明）。
  2. 授權 (Authorization)：確認你是誰之後，系統查表發現你的職位是「會計」。
  3. 權限管理/RBAC：因為你是「會計」（角色），所以你的卡片可以刷開「財務室」的門，但刷不開「IT機房」的門。

簡易流程圖

登入請求
→
身份驗證
(Authentication)
→
授權決策
(Authorization)
→
放行 或 拒絕
(授權決策依據：角色、層級、屬性等)
【第四步：選項分析】

本題四個選項皆正確，分析如下：

(A) 一旦使用者通過身份驗證後，系統會根據其授權層級與角色來分配相應的權限 [正確]

這是標準流程。必須先確認「你是誰」（身份驗證 Authentication，確認你是誰，例如密碼/OTP），才能決定「你能做什麼」（授權 Authorization，確認你能做什麼）。權限通常綁定在你的職位或角色上。

(B) 可以確保每個使用者僅能存取其需要的資源或功能 [正確]

這描述了資安核心原則之一：最小權限（Least Privilege，只給完成工作需要的最少權限）。良好的權限管理就是要做到這一點，不給多餘的權限。

(C) 能夠防止越權存取 [正確]

越權存取（Unauthorized Access，做了不該做、看了不該看的）是權限管理要解決的主要問題。例如，防止一般員工看到經理的薪資表。透過嚴格的權限控管，可以阻擋這種行為。

(D) 可以透過以角色為基礎的存取控制（Role-based access control，RBAC）實作 [正確]

RBAC（以角色為基礎的存取控制）是目前實作權限管理最常見、最有效率的方法。它避免了直接針對「每個人」設權限，而是針對「角色」設權限，管理起來更方便。

【觀念補充與延伸】

1. 核心觀念定位

這題屬於 IAM（Identity and Access Management，身分識別與存取管理）的範疇，是資安考試的高頻考點。

2. 重要定義

  • 權限管理 (Privilege Management)：管理使用者對系統資源（如檔案、資料庫、應用程式）的存取能力（讀、寫、執行、刪除）。
  • 角色 (Role)：角色（把工作性質分組，例如「會計」、「人資」、「一般員工」）。在 RBAC 中，權限是給「角色」的，使用者再被分配到「角色」。

3. 驗證 vs 授權 對照表

項目 身份驗證 (Authentication) 授權 (Authorization)
目的 確認使用者是誰 (Who you are) 決定使用者能做什麼 (What you can do)
時間點 先發生 驗證成功後發生
例子 輸入密碼、指紋辨識、OTP 讀取檔案、修改資料、進入管理後台

4. RBAC 角色映射示例

RBAC 的邏輯是：使用者 → 角色 → 權限。

使用者 (User) 角色 (Role) 權限 (Permissions)
Alice 管理員 (Admin) 新增用戶、刪除資料、讀取所有文件
Bob 工讀生 (Intern) 讀取公開文件 (不可刪除，不可新增)

5. 觸類旁通

  • 對比概念：ABAC (Attribute-Based Access Control)：以「屬性」為基礎。例如：「只有在『上班時間』且『在公司IP』才能存取」。比 RBAC 更靈活但更複雜。
  • 變化題型：題目若提到「員工離職或轉調部門，權限未移除」，這是在考權限回收與生命週期管理。
【記憶口訣】

「先驗身分再授權，
角色管權最安全，
最小權限防越權。」

【常見陷阱】
  • 陷阱一：混淆驗證與授權。
    記住：驗證是「進門卡」，授權是「房間鑰匙」。
  • 陷阱二：以為 RBAC 自動等於最小權限。
    RBAC 只是工具，如果管理員把「超級管理員」的角色隨便給所有人，那依然違反了最小權限原則。RBAC 必須配合正確的設定才有效。
【正確答案】

正確答案：A、B、C、D

一句話理由：四個選項分別描述了權限管理的流程(A)、原則(B)、目的(C)與實作方法(D)，皆為正確敘述。

【自我檢測】

透過以下題目測試你是否真的理解了：

Q1. 小明輸入帳號密碼登入系統，這個動作屬於資訊安全的哪一個階段？

答案：身份驗證 (Authentication)
解析：輸入帳密是為了證明「我是小明」，這是驗證身分的過程。

Q2. 關於 RBAC (Role-Based Access Control)，下列敘述何者錯誤？
(A) 權限是直接分配給每一個使用者的
(B) 權限是分配給角色的
(C) 使用者被分配到角色中
(D) 可以簡化權限管理

答案：(A)
解析：RBAC 的核心就是「不」直接把權限給人，而是給「角色」。若直接給人，那是 DAC 或 ACL 的概念，管理會很混亂。

Q3. 「只給予員工完成工作所需的最低限度權限」，這是什麼原則？

答案：最小權限原則 (Least Privilege)
解析：這是防止越權存取與減少資安災害範圍最重要的原則。
