風險管理 網站安全 題目

13. AAA 公司網站常受駭客入侵，網站風險問題一直居高不下，關於降低網站風險問題處理實務，下列做法何者較為正確？

查看詳細解析
解題思路

本題關鍵字：降低風險實務、檢測、修補

網站風險問題居高不下，代表現有防護不足。降低風險的最佳實務（Best Practice）必須包含「主動發現問題」與「預防問題發生」的流程。

  • 發現問題：透過弱點掃描（工具掃描）與黑箱檢測（模擬外部攻擊）找出已知與未知的漏洞。
  • 預防問題：透過原始碼審查（Code Review）在開發階段就修正邏輯與安全錯誤。
選項分析
  • (A) 錯誤： 違反最小權限原則（Least Privilege）。若帳號被盜，不區分讀寫權限將導致攻擊者能輕易刪除或竄改資料庫，擴大損害範圍。
  • (B) 正確： 定期掃描與檢測能持續找出漏洞，Code Review 則能從源頭減少弱點，是完整的風險管理循環（PDCA）。
  • (C) 錯誤： 這是典型的敏感資訊洩漏風險。備份檔（.bak, .old）常被駭客掃描下載，若留在正式主機上，等於直接把原始碼送給駭客分析。應使用版本控制系統（Git）管理。
  • (D) 錯誤： 縱深防禦（Defense in Depth）觀念不足。WAF 只能防禦應用層攻擊，無法修補作業系統（OS）底層漏洞。若 OS 不更新，駭客可繞過 WAF 利用 OS 漏洞取得主機控制權。
核心觀念：網站安全開發生命週期 (SSDLC)

單純依賴設備（如 WAF）是被動防禦。主動防禦需要將安全性整合到開發流程中：
1. 設計時：權限最小化、威脅建模。
2. 開發時：安全編碼、Code Review。
3. 測試時：弱點掃描、滲透測試（黑箱）。
4. 維運時：修補 OS/套件漏洞、監控。

自我檢測
如果公司只做弱點掃描但完全不做修補與追蹤，風險會下降嗎？為什麼？
不會。掃描只是在「找問題」（發現風險），如果不進行修補（Remediation）與後續追蹤驗證，漏洞依然存在，風險並未實質降低。
為什麼「最小權限」可以降低被駭後的損害？
當帳號僅擁有完成工作所需的最低權限時，即使該帳號被攻擊者竊取，攻擊者能做的破壞也被限制在該權限範圍內（例如只能讀取而無法刪除資料），從而降低整體損害。
WAF 能擋住所有入侵嗎？請舉一個可能繞過 WAF 的方向。
不能。WAF 主要防禦常見的 Web 攻擊（如 SQL Injection, XSS）。攻擊者可能透過作業系統漏洞（OS Vulnerabilities）、伺服器設定錯誤、內部人員威脅或直接對 API 邏輯進行攻擊來繞過 WAF。
