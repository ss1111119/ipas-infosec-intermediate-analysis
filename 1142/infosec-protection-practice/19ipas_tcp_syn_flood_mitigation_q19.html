某公司對外提供展示用的 Web 服務，供業務人員於公司外部進行產品展示。近期發現外部使用者無法正常連線，但公司內部上網與對外連線均正常。經分析網路側錄與連線紀錄後，判斷該展示服務遭受大量異常 TCP 連線請求，疑似為 TCP SYN Flood 類型的阻斷服務攻擊。

【題目】

針對 TCP SYN Flood 攻擊，下列何項應對策略可能無法有效緩解攻擊的影響？

【第一步：找關鍵字】

請圈出以下重點：

TCP SYN Flood 三向交握 半開連線 連線資源耗盡 無法有效

名師點評： 題目問的是「無法有效」，這是一題反向選擇題。你要找的是「做了也沒用」的選項，而不是找最好的防禦方法。

【第二步：白話翻譯】
高中生秒懂版：

「這就像是有一堆奧客（攻擊者）一直打電話進來餐廳訂位（SYN請求），店員接起電話說『好喔請問幾位？』（SYN-ACK），結果對方就掛掉電話不說話了。餐廳的訂位本（連線資源）上寫滿了這些假名字，真正的客人反而訂不到位。」

題目問：「下面哪一招，對於解決這種『假訂位癱瘓餐廳』的狀況，基本上是沒什麼用的？」

專有名詞解釋：
  • TCP SYN Flood： 惡意產生大量「只送出 SYN、不完成後續連線」的請求，塞爆伺服器的連線佇列。
  • SYN Cookies： 伺服器的一種聰明機制，「先不要把訂位寫在紙上」，而是給對方一個加密號碼牌（Cookie），等對方真的回來確認了，才分配資源。
【第三步：解題思路】

一句話核心： SYN Flood 是「傳輸層（Layer 4）」的連線攻擊，DNS 相關機制（Layer 7/基礎設施）無法直接處理單一 IP 被打爆的問題。

  1. 先確認攻擊類型：TCP SYN Flood 發生在 TCP 三向交握的第一階段。
  2. 找出防禦層級：我們需要針對 TCP 協定或流量清洗的防禦。
  3. 排除無效選項：DNS 反向代理主要處理「網域名稱解析」，當攻擊流量是直接針對 IP 發動 TCP 連線時，DNS 解析這一關根本擋不住，或者攻擊者早已解析完畢直接打 IP。
【第四步：選項分析】
  • (A) 啟用 DNS 反向代理 （正確答案：因為它無效）
    DNS 反向代理（或 DNS Proxy）處理的是客戶端查詢 Domain 對應 IP 的過程。雖然有些進階的 CDN 服務會結合 DNS 轉向，但純粹的「DNS 反向代理」無法阻止攻擊者對已經得知的 IP 發送大量 SYN 封包。SYN Flood 打的是伺服器的 TCP Stack，跟名字解析無關。
  • (B) 設定 TCP 連接速率限制（Rate Limiting）
    這是有用的。限制同一個來源 IP 在一秒內只能發送幾次 SYN 請求，可以大幅降低攻擊流量的衝擊。
  • (C) 啟用 SYN Cookies 機制
    這是最經典且有效的防禦。伺服器在收到 SYN 時不分配記憶體資源，而是透過雜湊計算回傳 Cookie，直到收到 ACK 驗證通過才建立連線，完美解決資源耗盡問題。
  • (D) 將展示服務部署到 CDN 上
    這非常有效。CDN 擁有巨大的頻寬與分散式節點，可以在邊緣（Edge）就幫你擋下大量的惡意連線，只把合法的流量導回你的源站（Origin）。
【觀念補充與延伸】

核心觀念定位： iPAS 防護實務「SYN Flood 原理與防禦」是必考題型。

對比概念：

  • 應用層防護 (Layer 7)： 針對 HTTP GET/POST Flood（如 WAF）。
  • 傳輸層防護 (Layer 4)： 針對 SYN/ACK/UDP Flood（如防火牆、SYN Cookies）。

DNS 攻擊通常是指 NXDOMAIN Flood 或 DNS Amplification，那才是 DNS 防護該做的事。

【記憶口訣】

「SYN 打連線，不打 DNS」

「半開連線滿出來，快用 Cookie 擋下來」

【常見陷阱】
  • ⚠️ 看到 "Proxy" 就覺得可以擋： 反向代理有分種類，HTTP 反向代理（如 Nginx）可以擋，但選項特別寫 "DNS" 反向代理，這就是陷阱。
  • ⚠️ 沒注意看題目問「無法」： 考試緊張時容易選成「最有效」的選項（很多人會選 C），結果痛失分數。
【正確答案】

(A)
