13. 有關「網頁應用程式防火牆 (WAF)」的描述，下列何者較為適切？

完整解析與觀念剖析

第一步找關鍵字

網頁應用程式防火牆（WAF）：題目主角，考你對 WAF 能力與限制的正確認知。
較為適切：代表不是「完美描述」，而是「最符合實務」的選項。
被繞過、無法完全阻擋：資安考試常見關鍵詞，暗示「沒有任何防護是 100%」。

第二步白話翻譯

「WAF 這個東西，在真實世界中，對網頁攻擊的防護能力，下列哪個說法比較接近事實？」

第三步解題思路

  • 1. 解題步驟：先確認 WAF 的定位是「降低風險」而非「消除風險」。接著檢查選項，若出現「只能」、「一定」、「完全」等絕對用語，通常有問題。
  • 2. 原理與規則：WAF 站在網站前面檢查 HTTP/HTTPS 請求，主要依賴特徵碼或行為分析。但攻擊手法會不斷變形（如編碼、分段），導致 WAF 可能漏接。
  • 3. 生活例子：WAF 就像大樓的「保全 ＋ 金屬探測門」，它可以擋掉大部份帶刀的人，但如果是高手把刀拆解藏在鞋底（變形攻擊），保全可能就抓不到。所以保全是重要的，但不能保證 100% 安全。

第四步選項分析

A 錯 定位錯誤：過濾垃圾郵件是「郵件安全閘道 (ESG)」的工作，不是 WAF。
B 對 符合實務：WAF 的規則確實可能被繞過（Bypassed），資安界公認無法完全阻擋所有攻擊。
C 錯 功能低估：WAF 不只防 Injection，還能防 XSS、Bot、路徑穿越等多種威脅。
D 錯 過時描述：現代 WAF 具備「正規化」能力，可還原變形特徵，並非只能比對一模一樣的字串。

觀念補充與延伸

法條或定義（白話）

WAF：一種專門保護 Web 應用程式，透過檢查 HTTP/HTTPS 流量來偵測與阻擋攻擊的安全機制。

觸類旁通（舉一反三）
  • 對比概念：WAF 是「補救與監控」，而 Secure Coding (安全程式設計) 才是根本解法。
  • 變化題型：若問「為什麼有 WAF 還要修補程式漏洞？」，答案就是因為「WAF 可能被繞過」。
記憶口訣
「WAF 能擋多，不能擋光」或「WAF 是防線，不是保證」。

常見陷阱

心理盲點：考試時看到「防火牆」三個字，直覺認為它很強大、無堅不摧，容易誤選「可以完全阻擋」的選項。

避坑指南：看到選項出現「完全」、「只能」、「一定」這種過度絕對的詞彙，先懷疑它。

自我檢測（點擊查看答案）
為什麼部署 WAF 之後，仍然要修補程式漏洞？
答案：因為 WAF 的規則有可能被繞過，若程式碼本身不修補，一旦 WAF 失效或被繞過，系統將直接暴露在風險中。
如果 WAF 停用或故障，哪一層防護最直接承受攻擊？
答案：Web 應用程式本身（程式碼層）。
WAF 最適合被放在縱深防禦的哪一層？
答案：應用層 (Application Layer) 防護的一環，需搭配安全程式設計與監控。
