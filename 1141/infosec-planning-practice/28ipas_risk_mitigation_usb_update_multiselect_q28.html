風險緩解：機台程式更新風險

28. 【複選題】
甲公司因 A 製程所用之所有機台當機，造成生產線停工、影響產品生產。經公司資安人員分析評估後，確認發生原因為廠商透過 USB 更新機台應用程式過程控制不當，導致機台程式異常所致。請問下列哪些選項為降低機台程式更新異常的風險緩解（Risk Mitigation：降低風險發生機率或影響程度）措施？

請注意題目中的重點標示：

  • 機台當機／生產線停工：這是已發生的後果。
  • USB 更新程式：這是問題發生的途徑。
  • 控制不當：這是根本原因。
  • 風險緩解（Risk Mitigation）：這是題目要求的解法。

解釋：本題重點在「降低更新過程出錯的機率或影響」，不是把風險轉走或不做。

用白話文來說：

工廠機台因為用 USB 更新程式出包，導致停工。現在要想辦法「讓以後更新比較不容易出事（降低機率）」或者「就算出事也不會全掛（降低影響）」，哪些做法真的能做到這兩點？

這題在考什麼觀念？
四種風險處理策略（避免、轉移、緩解、接受）中，「風險緩解」的實務判斷。

解題步驟：

  1. 判斷該措施是否「仍然繼續做 USB 更新」？（如果是停止不做，通常是避免）。
  2. 該措施是否能「降低出錯機率」或「縮小影響範圍」？
  3. 排除「避免風險（完全不做）」與「風險分擔（找人賠錢）」的類型。

原理與規則（白話）：

  • 風險緩解 (Risk Mitigation)：事情照做，但加上控制措施讓它比較不容易出事，或出事時比較不痛。
  • 不是不做（那是避免），也不是只談賠錢（那是分擔/轉移）。

(A) 將 A 製程委外處理
為什麼不是？委外等於自己不處理該製程，屬於「風險避免」或「風險轉移」，並沒有降低更新過程本身的技術風險。

(B) 要求 USB 使用前，應掃毒確認無異常
為什麼是？掃毒可以直接減少惡意程式或病毒透過 USB 進入機台的機會，這是在「降低發生機率」。

(C) 要求負責機台之廠商提供保證金
為什麼不是？收保證金只能彌補事後的財務損失，無法讓機台更新比較不出錯。這屬於「風險分擔（轉移）」。

(D) 要求機台需分多批次更新程式
為什麼是？分批更新（例如先更 10 台，沒事再更下一批）可以避免一次全部當機。這是典型的「降低影響範圍」。

選項 是否降低發生機率 是否降低影響範圍 風險處理類型
(A) 委外 否 否 風險避免 / 轉移
(B) 掃毒 是 否 風險緩解
(C) 保證金 否 否 (僅補償財務) 風險分擔 / 轉移
(D) 分批更新 否 是 風險緩解

核心觀念定位：

  • 資安管理 → 風險管理與風險處理
  • 考試常見度：必考（風險處理判定出現率極高）。

定義整理（白話）：

  • 風險緩解 (Risk Mitigation)：採取控制措施，降低風險發生的「機率」或「影響程度」。

觸類旁通（對比概念）：

  • 掃毒、分批更新、加裝防火牆 → 緩解
  • 委外不做、關閉功能 → 避免
  • 保險、合約求償、保證金 → 分擔 (或轉移)

「風險要緩解：先檢查、再分批；不做是避免，賠錢是分擔」

  • 易錯點：看到「保證金」或「罰款」就以為是風險降低。其實那只是錢的補償，機台還是會壞。
  • 心理盲點：把「降低財務損失」誤當成「降低資安事件發生」。
  • 避坑指南：問自己一句話：這個措施有沒有讓「出錯比較不容易發生」？如果有，就是緩解；如果只是事後拿錢，就是分擔。

正確答案：B、D

一句話理由：掃毒（B）與分批更新（D）能直接降低程式更新異常的發生機率或影響範圍。

請嘗試回答下列變形題，點擊標題查看答案：

1) 如果公司建立「更新前測試環境」先測試程式，屬於哪一種風險處理？
參考答案：風險緩解 (Risk Mitigation)。
理由：透過測試找出 Bug，降低實際上線出錯的機率。
2) 如果公司決定完全停止使用 USB，改用安全的集中更新系統，較接近哪一種？
參考答案：風險避免 (Risk Avoidance)。
理由：停止使用原本有風險的途徑（USB），就是避免掉該途徑的風險。
3) 如果公司購買營運中斷保險，較接近哪一種？
參考答案：風險分擔 (Risk Sharing) 或 風險轉移。
理由：涉及保險與金錢賠償，是將財務後果分擔給保險公司。
